Diagnóstico central

La medición tiene cobertura amplia, especialmente en GA4, registro, navegación, apuesta deportiva, Meta y Google Ads. Sin embargo, el contenedor muestra señales de crecimiento acumulativo: muchos tags por interacción, naming inconsistente, dependencia alta de selectores CSS y ausencia de clasificación explícita de consentimiento dentro de GTM.

La prioridad no debería ser crear más eventos, sino estabilizar las conversiones críticas, limpiar el modelo de datos y convertir la medición en un sistema gobernable para pauta, analítica y producto.

Lectura ejecutiva del diagnóstico

Riesgo alto

Consentimiento y datos personales

Todos los tags aparecen con consentimiento no configurado en GTM. Además, existe un HTML asociado a registro avanzado que captura email/teléfono, mientras Google Ads tiene conversiones avanzadas desactivadas.

  • Acción inmediata: confirmar si la CMP controla el disparo antes de GTM o si se requiere configurar Consent Mode en el contenedor.
  • Definir si Enhanced Conversions se implementa bien o se retira la captura de datos.
Riesgo alto

Conversiones principales sin validación funcional

El JSON muestra tags para registro completo, apuesta exitosa, login y remarketing, pero no confirma que disparen sin duplicidad ni que lleguen correctamente a cada plataforma.

  • Acción inmediata: prueba end-to-end con GTM Preview, GA4 DebugView, Network, Meta y Google Ads.
  • Primero validar registro completo y apuesta exitosa; después interacciones secundarias.
Riesgo medio

Modelo GA4 atomizado

Hay 321 tags GA4 Event y 54 nombres únicos de evento. Muchos nombres mezclan dispositivo, sección y acción en el nombre del evento, lo que complica reportes y evolución del modelo.

  • Acción: migrar gradualmente a eventos canónicos con parámetros.
  • Menos eventos, más contexto en parámetros.
Riesgo medio

Fragilidad por selectores CSS

Hay 323 triggers con CSS_SELECTOR. Esta estructura funciona mientras el frontend no cambie; si cambia una clase, se puede caer la medición sin aviso.

  • Acción: pasar eventos críticos a dataLayer.
  • Mantener clicks CSS solo para tracking secundario o exploratorio.

Modelo de eventos sugerido

La mejora más importante para análisis no es medir más clics, sino crear un lenguaje común de medición. Esta taxonomía permite mantener la riqueza del dato sin multiplicar tags por cada botón.

Decisiones que necesita tomar el equipo

1. ¿Qué será conversión principal?

Definir si optimización paid se enfocará en registro completo, primer depósito, apuesta exitosa o una combinación jerarquizada.

2. ¿Cómo se tratará el valor de apuesta?

Decidir si el valor enviado a Meta/GA4 representa stake, revenue, GGR estimado u otro concepto. Esto evita lecturas infladas de ROAS.

3. ¿Se implementarán Enhanced Conversions?

Si la respuesta es sí, debe hacerse con consentimiento, hashing y configuración formal. Si no, retirar captura heredada.

4. ¿Qué eventos se migran a dataLayer?

Priorizar registro, login, apuesta, depósito y errores críticos. Los clics secundarios pueden esperar.

Qué no debe hacerse de inmediato

Recomendación operativa

Evitar limpiar todo sin validar primero

No conviene eliminar tags o renombrar eventos masivamente sin antes tener una medición base validada. El primer objetivo es asegurar continuidad de conversiones y atribución; después se depura y estandariza.
